# A/B тест: новый экран онбординга — влияние на конверсию и retention

**Стек:** Python · pandas · SciPy · statsmodels · matplotlib
**Тип:** портфолио-проект (синтетические данные)

---

## Бизнес-контекст

Мобильное FinTech-приложение планирует раскатить редизайн экрана онбординга. До полной раскатки продуктовой команде нужно ответить на три вопроса:

1. **Увеличивает** ли новый флоу конверсию signup → first action на бизнес-значимую величину?
2. Не **вредит** ли он раннему удержанию (Day-7 retention)?
3. Работает ли эффект во всех сегментах или тянется одной когортой?

Проект симулирует полный пайплайн, который аналитик прошёл бы в реальности — от дизайна теста до решения о раскатке — на синтетических данных.

---

## Методология 

| Шаг | Зачем |
|-----|-------|
| **Power analysis до запуска теста** | Размер выборки посчитан из baseline CR, MDE, α и power — а не взят «на глаз» |
| **Проверка Sample Ratio Mismatch (SRM)** | Ловит баги бакетинга до того, как они обнулят анализ |
| **Сравнимость групп до теста** | Проверяем, что распределение по неделям регистрации сбалансировано |
| **Размер эффекта + 95% CI на разницу** | Бизнес принимает решения по величине лифта, а не по p-value |
| **Guardrail-метрика (retention)** | Не даёт зашипить «победу», где просто протолкнули низкокачественных юзеров |
| **Сегментный анализ** | Показывает, не скрывает ли средний эффект вред какой-то подгруппе |
| **Структурированное решение: ship / hold / kill** | То, что реально нужно стейкхолдерам |

## Дизайн теста

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Baseline конверсия | 12.0% | По историческим данным (принимаем как есть) |
| Minimum Detectable Effect (MDE) | +2 п.п. абсолютных | Ниже — инженерные затраты не оправданы |
| Alpha | 0.05 | Стандартный двусторонний порог |
| Power | 0.80 | Стандарт для продуктовых экспериментов |
| Необходимый n на группу | 3 492 | Из `NormalIndPower.solve_power()` |
| Фактический n на группу | 5 000 | Запас для сегментного анализа |

**Гипотезы**
- H0: конверсия(тест) = конверсия(контроль)
- H1: конверсия(тест) > конверсия(контроль)

**Основная метрика:** конверсия signup → first_action (бинарная на пользователя)
**Guardrail:** Day-7 retention (не должна упасть)

---

## Результаты

| Метрика | Контроль | Тест | Абс. лифт | 95% CI на разницу | p-value |
|---------|----------|------|-----------|-------------------|---------|
| Конверсия | 11.56% | 15.06% | +3.50 п.п. | [+2.17, +4.83] | < 0.0001 |
| Retention D7 | 11.40% | 11.30% | −0.10 п.п. | не значимо | 0.87 |

**Сегментный анализ** подтвердил: лифт есть во всех четырёх когортах по неделям регистрации (диапазон от +1.4 до +5.7 п.п.), то есть эффект не тянется одной подгруппой.

**SRM check:** разбиение 50/50 не поехало (chi-square p = 1.00) — багов бакетинга нет.

---

## Решение

** SHIP** — раскатывать на 100%.

Обоснование: основная метрика выросла на бизнес-значимую и статистически значимую величину, guardrail держится, эффект стабилен во всех сегментах.

---

## Что проект НЕ делает (и почему)

- **Нет мониторинга в реальном времени / sequential testing.** В реальных экспериментах нужны правила ранней остановки; здесь тест взят как fixed-horizon для ясности.
- **Нет байесовского анализа.** Взят частотный подход — так работает большинство продуктовых команд.
- **Нет поправки на множественные сравнения.** С одной основной и одной guardrail-метрикой family-wise error приемлема; для полного набора метрик пришлось бы применять Бонферрони или контроль FDR.

Это осознанные ограничения области, а не пропуски — готов обсуждать любое из этих решений подробнее.
